三、严重缺陷详析

缺陷 1：建禄格/月刃格完全缺失

项目内容
典籍规定月令为比劫时，月令本身不可取用，到四柱另寻财官食印透干会支者取用。建禄一般身旺喜克泄耗（官杀/食伤/财星）；月刃为凶神必须官杀制刃或食伤泄刃
代码现状_TG_TO_PATTERN 无"比肩"和"劫财"映射 → determine_pattern_type() 返回空字符串 → PATTERN_YONGSHEN.get("", []) 返回空列表 → 格局法零候选
影响面任何月令本气十神为比肩或劫财的命盘。例如：甲日主生寅月（比肩）、乙日主生卯月（比肩）、甲日主生卯月（劫财）等。这在实际八字中占比约 12-15%
当前兜底只能靠扶抑法/调候法/通关法生成候选，但这些候选没有格局法加持（weight=2.0），初始排序分会很低

缺陷 2：从弱格/专旺格扶抑方向错误

项目内容
典籍规定从弱格：顺弱势，用官杀/财/食伤（忌印比）；专旺格：食伤泄秀为首选，次取财星（忌官杀犯旺）
代码现状从弱格 → judge_wangshuai_level 返回"极弱" → direction="生扶" → 扶抑法生成正印/偏印/比肩/劫财 → 正好是典籍禁忌的印比
专旺格 → 返回"极旺" → direction="克泄耗" → 扶抑法生成正官/七杀 → 正好是典籍禁忌的官杀
额外问题PATTERN_YONGSHEN 无"从弱格"/"专旺格"条目 → 格局法零候选 → 扶抑法成为唯一来源，且方向错误 → 候选完全错误

缺陷 3：调候法只覆盖冬夏两季

项目内容
典籍规定《穷通宝鉴》有完整的 10日干×12月令 调候用神矩阵。每个月令都有具体的调候用神，不仅限于冬夏
代码现状_get_tiaohou_wuxing() 只处理：冬月(亥子丑)→火、夏月(巳午)→水。春秋8个月完全无调候候选
影响春季(寅卯辰)和秋季(申酉戌)出生的命盘不会生成调候法候选。典籍中寅月需丙火照暖、辰月需甲木疏土、申月需丁火炼金等——这些都是重要的用神来源

四、中等问题

问题 4：用神优先级与典籍不符

优先级典籍代码 weight差异
第一（极端气候）调候法格局法(2.0)代码格局法始终最高，但典籍说极端气候调候优先
第二通关法调候法(1.5) = 通关法(1.5)通关法与调候法同级，问题不大

说明：典籍的优先级是"以急缓定主次"——极端气候调候优先，非极端格局优先。代码的固定 weight 无法体现这种动态优先级。但这在混合架构中可以由 LLM 来判断"是否极端气候"。

六、对混合架构的影响

可以继续推进的

  • 八格（正格）候选生成逻辑正确——PATTERN_YONGSHEN 的设计思路验证通过
  • 4来源架构（格局/调候/通关/扶抑）符合典籍体系——典籍确实有这四种取用方法
  • 通关法5组五行对应正确——代码的 _TONGGUAN_MAP 与典籍一致
  • 扶抑法身旺/身弱方向正确（仅对正格而言）

必须先修复的

  • P0：建禄格/月刃格——12-15% 的命盘完全没有格局法候选，LLM 再强也没有正确的候选可选
  • P0：从弱格/专旺格方向——扶抑法生成的候选与典籍完全相反，会误导 LLM
  • P1：调候法覆盖——60% 的命盘缺少调候候选，少了一个重要的用神来源

可以在混合架构中由 LLM 弥补的

  • 用神优先级的动态判断（极端气候 vs 非极端）——LLM 可以根据月令判断是否需要调候优先
  • 偏印格缺食神——LLM 可以在候选生成后补充
  • confidence 递减——LLM 可以通过问题设计体现候选优先级
