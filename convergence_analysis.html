验证流程逐轮拆解 — 收敛问题诊断报告

对照 services/verification.py 实际代码，逐轮分析每一轮问答的后台状态、权重调整、收敛条件，以及 LLM 介入后哪里断裂

一、全局架构：两条路径

代码里有两条执行路径，取决于环境变量 DEEPSEEK_API_KEY 是否存在：

路径 条件 执行逻辑 收敛状态
V3 静态 无 API Key _dispatch_static() 按状态机执行：每轮回答→调 confidence→检查收敛→推进阶段 正常收敛
V4 LLM 有 API Key _llm_next() 先拦截：LLM 决定 ask_question/advance_stage/backtrack。只有 LLM 返回 None 时才降级到静态 无法收敛

根本问题：LLM 拦截点与静态状态机的断裂

_llm_next() 第 495 行的拦截规则：

if sub.endswith("_L3") or sub in ("ys_3", "ys_2"): return None # 只有这些阶段才交给静态状态机

也就是说，LLM 在以下阶段全部拦截：

  • L1 — 格局特征验证
  • phase2_L2 — 有情验证
  • diag_D1 ~ diag_D4 — 诊断链
  • ys_1 — 用神第一轮

只有 phase2_L3、ys_2、ys_3 才降级到静态。

当 LLM 返回 ask_question 时，sub_stage 不变，下一轮还是被 LLM 拦截 → 无限循环

例（七杀格+身旺）："你人生中是否经历过较大的压力或挑战，但事后回头看，那些压力反而促成了你的成长？"

V4 LLM 路径：实际发生了什么

BUG LLM 通过 _llm_next() 生成问题，返回 action: "ask_question"

后果：

  • confidence 不变 — LLM 返回的 confidence 值存在 session["_llm_last_quality"]，但没写回 session["confidence"]
  • sub_stage 不变 — 仍然是 "L1"，下一轮还是被 LLM 拦截
  • l1_answer 不设置 — _handle_L1() 从未执行
  • 无推进计数器 — 没有任何机制限制 LLM 在同一阶段问几轮

结果：LLM 在 L1 阶段反复生成问题，永远不返回 advance_stage

每一步都是问用户一个 yes/no 问题，"是"就回品质评估，"否"就下一步。

特殊规则：diagnosis_count（否定次数）≥ 3 → 直接跳 D5

用神收敛阶段：ys_1 → ys_2 → ys_3（核心收敛逻辑）

候选生成（进入用神阶段时一次性生成）

以 七杀格 + 身旺（方向=克泄耗） 为例：

来源 权重 weight 初始 confidence 初始排序分 (conf×weight) 候选用神
格局法 2.0 35 70 食神(35), 正印(35), 伤官(35)

扶抑法 1.0 15 15 身旺→克泄耗方向

收敛条件检查（每轮回答后都检查）

条件 1（提前锁定）：top.confidence >= 65 且 top - second >= 20

条件 2（强制结束）：sub_stage == "ys_3" → 无论是否满足条件 1，直接 finalize

条件 3（再生）：如果 next_idx > 3 且 regeneration < 2 → regeneration++，top.confidence += 5，重置到 ys_1

V4 LLM 路径：用神阶段发生了什么

BUG ys_1 被 LLM 拦截（_llm_next() 第 495 行只放行 ys_2 和 ys_3）

当 LLM 返回 action: "ask_question" 时：

  • _handle_yongshen() 从未执行 — 以下代码全部跳过：
  • - confidence +15/+3/-20 调整 → 不执行
  • - 候选重新排序 → 不执行
  • - 收敛条件检查 → 不执行
  • - sub_stage 推进到 ys_2 → 不执行
  • sub_stage 仍然是 "ys_1" → 下一轮还是被 LLM 拦截 → 无限循环

即使 LLM 正确生成了关于食神的问题，用户的回答也完全不会影响候选的 confidence。

候选的 confidence 永远停留在初始值（35/25/22/15），永远不满足 65 的收敛阈值。

三、收敛断裂点总结

5 个断裂点

#位置V3 静态（正常）V4 LLM（实际）后果
1 L1 格局验证 回答→调 confidence→设 l1_answer→推进 LLM 生成问题，不调 confidence，不设 l1_answer，不推进 卡在 L1 无限问
2 phase2_L2 有情 回答→设 phase2_youqing→推进 L3 LLM 生成问题，不设 youqing，不推进 卡在 L2 无限问
3 diag_D1~D4 诊断链 回答→设 diagnosis_path→推进或回 Phase2 LLM 生成问题，不推进诊断链 卡在诊断步无限问
4 ys_1 用神第一轮 回答→调候选 confidence→重排→检查收敛→推进 ys_2 LLM 生成问题，不调 confidence，不检查收敛，不推进 卡在 ys_1 无限问，候选分数永远不变
5 初始 confidence 过低 身旺→30，需要至少2轮"很像"才能到65 即使 V3 正常，3轮问3个不同候选，单个候选最多被问1次，很难到65 靠 ys_3 强制结束，confidence 可能只有50-56

四、为什么 LLM 不返回 advance_stage

LLM 的 prompt（第 504-524 行）告诉它：

规则: - 如果当前格局特征已足够确认，action="advance_stage" - 如果用户持续否定，可以 action="backtrack"

但存在以下问题：

  • "足够确认"没有量化标准 — LLM 不知道 confidence 是多少，也不知道阈值是 65
  • LLM 看不到候选列表 — prompt 里没有传入 yongshen_candidates 的 confidence 值
  • 没有轮次限制 — prompt 没说"你最多问 3 轮必须 advance"
  • LLM 倾向于继续追问 — 命理师的 system prompt 鼓励深入验证，所以 LLM 自然倾向于多问

五、修复方向（仅供参考，不改动代码）

方案 A：LLM 只增强问题文本，不控制流程

把 _llm_next() 改为：先执行静态 handler 拿到问题和阶段推进，再用 LLM 增强问题文本。LLM 不决定 action。

方案 B：给 LLM 加轮次硬限制

在 _llm_next() 里加计数器：同一 sub_stage 最多问 2 轮，第 3 轮强制 advance_stage。

方案 C：LLM 返回 ask_question 时也执行权重调整

在 ask_question 分支里，先调用静态 handler 做 confidence 调整，再用 LLM 的问题文本覆盖。

方案 D：降低收敛阈值 或 改为相对排序

把 65 降到 50，或者改为"3 轮后取 confidence 最高的"（实际上 ys_3 已经这样做了，但 LLM 路径走不到 ys_3）。
